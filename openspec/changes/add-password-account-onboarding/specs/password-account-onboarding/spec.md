## 新增需求

### 需求:系统必须在首次安装时为管理员提供密码设置能力
当实例尚未创建任何 workspace 且 password auth 已启用时，系统必须允许管理员在首次安装表单中直接设置密码，并在安装成功后得到一个已具备密码凭据的管理员账号。

#### 场景:首次安装时成功创建带密码的管理员账号
- **当** 用户在首次安装页面提交合法的 `teamName`、`userName`、`userEmail`、`password` 和确认密码，且当前实例还没有任何 team
- **那么** 系统必须创建 workspace、创建管理员用户、写入密码哈希，并完成登录态建立

#### 场景:首次安装缺少密码字段时拒绝创建
- **当** 首次安装请求缺少密码、确认密码，或两者不匹配
- **那么** 系统必须返回校验错误，且不得创建 team 或用户

#### 场景:已有 team 时仍禁止重新安装
- **当** 实例中已经存在至少一个 team
- **那么** 系统必须继续拒绝 `installation.create`，即使请求中包含合法密码字段

### 需求:系统必须允许受邀成员通过一次性激活链接首次设置密码
当 workspace 启用了 password auth 且管理员邀请了一个新成员时，系统必须向该受邀成员提供一次性激活链接，用于首次设置密码并完成账号激活。

#### 场景:password auth 启用团队邀请新成员时发放激活链接
- **当** 管理员在启用了 password auth 的 team 中邀请一个此前不存在的成员邮箱
- **那么** 系统必须创建 invite shell user，并生成一枚一次性激活 token 供该成员设置首次密码

#### 场景:password auth 未启用团队保持原邀请路径
- **当** 管理员在未启用 password auth 的 team 中邀请成员
- **那么** 系统必须保持原有邀请行为，不得要求受邀成员通过 password 激活

#### 场景:重复邀请已存在未激活用户时不得生成多个有效激活入口
- **当** 一个邮箱已经对应当前 team 的 invite shell user，且尚未完成首次激活
- **那么** 系统必须避免并存多个可无限使用的激活入口，并保持激活语义仍然是一位受邀成员对应一个待激活账号

### 需求:系统必须在首次设密码成功后将受邀成员标记为已激活账号
受邀成员通过激活链接首次设置密码时，系统必须把该账号从 invited/shell 状态转换为可登录账号，并只允许该激活语义成功一次。

#### 场景:激活链接首次使用成功
- **当** 受邀成员提交合法且未消费的激活 token 与满足密码策略的新密码
- **那么** 系统必须写入密码哈希、更新账号激活状态、记录首次激活审计信息，并使该账号具备后续 password 登录能力

#### 场景:激活 token 被重复使用
- **当** 同一激活 token 已被成功消费后再次提交
- **那么** 系统必须拒绝本次激活请求，且不得重复修改密码或重复触发账号激活副作用

#### 场景:已激活账号再次使用激活链接
- **当** 目标用户已经不是 invite shell user，或已经拥有可用密码凭据
- **那么** 系统必须拒绝把该请求当作首次激活处理

### 需求:系统必须保持 password onboarding 不等同于公开注册
password onboarding 只允许发生在“首次安装创建管理员”和“已有邀请记录的成员激活”两个场景，系统不得因此开放面对任意访客的公开 password 注册能力。

#### 场景:/create 页面仍不展示公开 password 注册入口
- **当** 未登录访客访问 `/create`
- **那么** 系统不得把 password provider 当作公开自助注册入口展示给任意访客

#### 场景:未受邀用户不得通过激活路径创建账号
- **当** 某个邮箱不存在对应 team 的邀请用户记录
- **那么** 系统不得允许该邮箱仅凭 password onboarding 路径直接创建新成员账号
