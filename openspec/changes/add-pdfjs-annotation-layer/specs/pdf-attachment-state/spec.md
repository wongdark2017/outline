## 修改需求

### 需求:系统必须保存 PDF attachment 侧车状态
系统必须为 PDF attachment 提供 Outline 内部侧车状态，状态绑定到当前 document 和 attachment，而不是写入 PDF 文件本体或 ProseMirror 节点 attrs。系统必须将正式用户可见的 PDF 批注状态保存为 PDF.js 页内标注数据，不再保存或展示临时纯文本 note 面板作为正式能力。

#### 场景:无状态时返回空状态
- **当** 有 document read 权限的用户请求一个尚未保存批注状态的 PDF attachment
- **那么** 系统返回 `revision = 0` 和空 PDF 标注状态

#### 场景:保存新页内标注状态
- **当** 有 document update 权限的用户提交 `revision = 0` 和合法 `version = 2` 页内标注数据
- **那么** 系统创建 `AttachmentPdfState` 记录并返回 `revision = 1`

#### 场景:更新已有页内标注状态
- **当** 有 document update 权限的用户提交当前 revision 和合法 `version = 2` 页内标注数据
- **那么** 系统更新同一 `(documentId, attachmentId)` 状态并返回递增后的 revision

#### 场景:状态不写入 PDF 文件
- **当** 用户保存 PDF 页内标注状态
- **那么** 系统不得修改 attachment 原文件、不得上传新 PDF、不得更新 ProseMirror attachment 节点的 `id` 或 `href`

#### 场景:状态不写入节点 attrs
- **当** 用户保存 PDF 页内标注状态
- **那么** 系统不得把批注 JSON 写入 ProseMirror attachment 节点 attrs

#### 场景:读取旧版纯文本状态
- **当** 系统读取到旧 `version = 1` 纯文本 note 状态
- **那么** 系统禁止将其作为正式页内标注渲染到 PDF 页面上

### 需求:系统必须校验 PDF 状态数据
系统必须校验 PDF 状态 JSON 的结构和大小，防止无效数据或过大数据写入数据库。系统必须支持 `version = 2` 页内标注状态，并校验每条标注的类型、模式、颜色、页码、归一化 rects 和文本长度。

#### 场景:接受 version 2 状态
- **当** 用户提交 `version = 2` 且结构合法的 PDF 页内标注状态
- **那么** 系统允许保存该状态

#### 场景:拒绝无效版本
- **当** 用户提交既不是兼容读取版本也不是当前写入版本的 PDF 状态
- **那么** 系统拒绝请求

#### 场景:拒绝无效页码
- **当** 用户提交 `pageIndex` 小于 0 或不是整数的标注
- **那么** 系统拒绝请求

#### 场景:拒绝无效标注类型
- **当** 用户提交不在 `text`、`rectangle` 范围内的 version 2 标注类型
- **那么** 系统拒绝请求

#### 场景:拒绝无效标注模式
- **当** 用户提交不在 `highlight`、`fill`、`border` 范围内的标注模式
- **那么** 系统拒绝请求

#### 场景:拒绝无效颜色
- **当** 用户提交不是允许颜色格式的标注颜色
- **那么** 系统拒绝请求

#### 场景:拒绝无效归一化 rect
- **当** 用户提交 rect 坐标、宽度或高度不是有限数值，或超出服务端允许范围
- **那么** 系统拒绝请求

#### 场景:拒绝过长文本
- **当** 用户提交超过服务端限制长度的标注文本或选中文本
- **那么** 系统拒绝请求

#### 场景:拒绝过多标注
- **当** 用户提交超过服务端允许数量的标注列表
- **那么** 系统拒绝请求

### 需求:前端必须以 PDF.js 页内方式展示和保存 PDF 批注状态
前端必须使用 PDF.js 阅读器展示和保存 PDF 页内标注状态。前端禁止继续展示当前纯文本 note sidecar 面板作为 PDF 批注入口。

#### 场景:读取 PDF 页内标注状态
- **当** 用户打开带有 attachment id 的 PDF attachment 阅读视图
- **那么** 前端请求 `attachments.pdfState.get` 并将返回的 version 2 标注渲染到 PDF 页面标注层

#### 场景:保存 PDF 页内标注状态
- **当** 有 update 权限的用户创建、修改或删除页内标注并保存
- **那么** 前端调用 `attachments.pdfState.update` 并带上当前 revision 和 version 2 状态

#### 场景:只读用户不显示标注编辑入口
- **当** 用户只能读取文档
- **那么** 前端必须展示已有页内标注但不显示标注创建、删除或保存入口

#### 场景:缺少 attachment id 时不保存
- **当** PDF attachment 节点缺少 attachment id 且无法从 href 恢复 attachment id
- **那么** 前端必须允许查看 PDF 原件但不显示 PDF 状态保存入口

#### 场景:移除纯文本批注面板
- **当** PDF 阅读态渲染
- **那么** 前端禁止展示仅包含 textarea 的纯文本 `Annotations` 创建面板作为 PDF 批注功能

### 需求:系统必须明确 PDF 侧车状态可见性
系统必须把 PDF 批注状态定义为 Outline 内部页内标注状态，并禁止向用户暗示原始 PDF 文件已被修改。

#### 场景:下载原 PDF 不包含侧车标注
- **当** 用户直接打开或下载原始 attachment
- **那么** 系统返回原 PDF 文件且不承诺包含 Outline 页内标注

#### 场景:UI 文案区分标注保存和 PDF 保存
- **当** 前端展示保存操作
- **那么** 文案必须表达保存的是标注状态，而不是保存或导出 PDF 文件本体

#### 场景:禁止写回 PDF
- **当** 用户保存或删除 PDF 标注
- **那么** 系统禁止修改原 PDF attachment 的文件内容
