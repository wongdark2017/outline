## 新增需求

### 需求:系统必须使用 PDF.js 渲染全屏 PDF 阅读态
系统必须在全屏 PDF 阅读态中使用 PDF.js 渲染 PDF 页面，并禁止依赖浏览器原生 PDF iframe 作为主阅读区。

#### 场景:打开 PDF.js 阅读态
- **当** 用户打开 PDF attachment 的全屏阅读态
- **那么** 系统必须通过 PDF.js 加载 `/api/attachments.file?id=<attachmentId>` 并渲染 PDF 页面

#### 场景:不使用 iframe 主阅读区
- **当** PDF 阅读态渲染
- **那么** 主 PDF 阅读区域禁止使用浏览器原生 PDF iframe 或 object 作为唯一阅读器

#### 场景:保留原文件打开入口
- **当** 用户点击打开原文件操作
- **那么** 系统必须仍然打开或下载原始 PDF attachment

#### 场景:PDF 加载失败
- **当** PDF.js 无法加载或解析 PDF
- **那么** 系统必须显示错误状态并保留打开原文件入口

### 需求:系统必须提供 PDF 页面导航和缩放
系统必须允许用户在 PDF.js 阅读态中查看多页 PDF、跳转页面并缩放页面。

#### 场景:渲染多页 PDF
- **当** PDF 包含多个页面
- **那么** 系统必须按页面顺序渲染并允许用户滚动查看所有页面

#### 场景:页面跳转
- **当** 用户输入或选择目标页码
- **那么** 系统必须滚动到对应 PDF 页面

#### 场景:缩放页面
- **当** 用户调整缩放级别
- **那么** 系统必须重新渲染页面并保持已保存标注与页面内容对齐

### 需求:系统必须支持文本高亮标注
系统必须允许有编辑权限的用户在 PDF text layer 中选择文本并创建页内高亮标注。

#### 场景:创建文本高亮
- **当** 有编辑权限的用户在单个 PDF 页面中选择文本并选择高亮颜色
- **那么** 系统必须创建 text 类型标注并在选中文本位置渲染高亮覆盖层

#### 场景:保存文本高亮内容
- **当** 用户创建文本高亮标注
- **那么** 系统必须保存标注页码、颜色、选中文本和归一化 rects

#### 场景:禁止跨页单条文本高亮
- **当** 用户选择范围跨越多个 PDF 页面
- **那么** 系统必须拒绝创建单条跨页高亮或将其拆分为多条单页标注

#### 场景:只读用户不能创建文本高亮
- **当** 只读用户在 PDF 中选择文本
- **那么** 系统禁止显示创建标注的编辑操作

### 需求:系统必须支持矩形区域标注
系统必须允许有编辑权限的用户在 PDF 页面上拖拽创建矩形区域标注。

#### 场景:创建填充矩形标注
- **当** 用户选择填充矩形模式并在 PDF 页面拖拽
- **那么** 系统必须创建 rectangle 类型 fill 模式标注并在页面上渲染半透明区域

#### 场景:创建边框矩形标注
- **当** 用户选择边框矩形模式并在 PDF 页面拖拽
- **那么** 系统必须创建 rectangle 类型 border 模式标注并在页面上渲染边框区域

#### 场景:保存矩形坐标
- **当** 用户完成矩形标注
- **那么** 系统必须保存该标注页码、颜色、模式和归一化 rects

#### 场景:只读用户不能创建矩形标注
- **当** 只读用户查看 PDF 页面
- **那么** 系统禁止进入矩形标注创建模式

### 需求:系统必须按页面坐标稳定渲染标注
系统必须使用页面归一化坐标保存和渲染 PDF 标注，避免缩放或窗口变化后标注漂移。

#### 场景:缩放后标注对齐
- **当** 用户改变 PDF 缩放级别
- **那么** 系统必须根据归一化坐标重新计算标注位置并保持与页面内容对齐

#### 场景:窗口尺寸变化后标注对齐
- **当** 阅读态宽度变化导致页面显示尺寸变化
- **那么** 系统必须重新布局标注覆盖层并保持与页面内容对齐

#### 场景:保存时不使用屏幕像素
- **当** 系统保存 PDF 标注状态
- **那么** 系统禁止将当前屏幕像素作为唯一坐标来源持久化

### 需求:系统必须提供标注颜色和模式工具栏
系统必须在 PDF.js 阅读态中提供标注工具栏，允许用户选择文本高亮、矩形模式和颜色。

#### 场景:选择颜色
- **当** 用户选择一种标注颜色
- **那么** 后续创建的文本或矩形标注必须使用该颜色

#### 场景:切换标注模式
- **当** 用户在文本高亮、填充矩形和边框矩形模式之间切换
- **那么** 系统必须只启用当前模式对应的创建交互

#### 场景:只读模式隐藏编辑工具
- **当** 用户没有文档更新权限
- **那么** 系统必须隐藏或禁用标注创建工具

### 需求:系统必须提供标注列表和跳转
系统必须展示当前 PDF 的标注列表，并允许用户点击标注跳转到对应页面和位置。

#### 场景:展示标注列表
- **当** PDF 状态中存在标注
- **那么** 系统必须在阅读态中展示标注列表，包含页码、类型、颜色和文本摘要

#### 场景:点击标注跳转
- **当** 用户点击标注列表中的一项
- **那么** 系统必须滚动到该标注所在页面并突出显示对应标注

#### 场景:删除标注
- **当** 有编辑权限的用户删除一个标注
- **那么** 系统必须从页面覆盖层和保存状态中移除该标注

### 需求:系统必须懒加载 PDF.js 阅读器
系统必须只在用户打开 PDF 阅读态时加载 PDF.js 相关代码，避免影响普通文档加载。

#### 场景:普通文档不加载 PDF.js
- **当** 用户打开不包含全屏 PDF 阅读态的文档
- **那么** 系统禁止主动加载 PDF.js 主包或 worker

#### 场景:打开 PDF 时加载 PDF.js
- **当** 用户打开 PDF 全屏阅读态
- **那么** 系统必须加载 PDF.js 阅读器代码和 worker

### 需求:系统必须支持 PDF.js worker 构建和安全策略
系统必须确保 PDF.js worker 在开发和生产环境中可加载，并且只添加最小必要的 CSP 配置。

#### 场景:开发环境加载 worker
- **当** 开发环境打开 PDF.js 阅读态
- **那么** PDF.js worker 必须成功加载并渲染 PDF 页面

#### 场景:生产构建加载 worker
- **当** 生产构建打开 PDF.js 阅读态
- **那么** PDF.js worker 必须从构建产物或允许的同源地址成功加载

#### 场景:禁止恢复 EmbedPDF 静态路由
- **当** 配置 PDF.js worker
- **那么** 系统禁止恢复 `/embedpdf/*` 静态资源路由
