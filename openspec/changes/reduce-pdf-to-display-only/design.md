## 上下文

`add-pdfjs-annotation-layer` 已经把独立 PDF 阅读态推进到了 `PDF.js + text layer + annotation layer + AttachmentPdfState` 的形态，但产品方向现在收缩为“只支持 PDF 显示”。这意味着需要从已经完成的批注方案中回退到更窄的阅读范围，同时避免把现有实现中的批注入口继续暴露给用户。

当前系统里有两条 PDF 路径：

- 文档内嵌 PDF 预览仍然使用轻量 `embed` 预览。
- 独立打开的 PDF 阅读界面使用 PDF.js 渲染页面和基础阅读控制。

回退方案需要在不破坏 PDF 打开、缩放、翻页和原文件访问的前提下，移除批注交互和与之绑定的状态读写。

## 目标 / 非目标

**目标：**

- 保留 PDF 显示能力，包括文档内嵌预览和独立阅读界面。
- 保留独立阅读界面的 PDF.js 页面渲染、导航、缩放和打开原文件入口。
- 移除所有用户可见的批注交互、批注列表、保存按钮和批注状态消息。
- 使 PDF 显示流程不再依赖 `attachments.pdfState.get/update`、`AttachmentPdfState` 或批注 sidecar 数据。
- 为未来重新讨论批注能力保留清晰边界，但不在当前版本中承诺或暴露。

**非目标：**

- 不实现任何 PDF 批注、保存或导出能力。
- 不把批注结果写回 PDF 文件本体。
- 不把文档内嵌 PDF 预览升级为批注感知组件。
- 不在本次回退中删除历史 `AttachmentPdfState` 数据记录，除非实现阶段确认完全无风险。
- 不重新设计新的 PDF 产品形态；当前只收缩到显示模式。

## 决策

### 决策 1：保留 PDF.js 独立阅读器，但将其收缩为 display-only

独立阅读界面继续使用 PDF.js，而不是完全回退到浏览器原生 iframe。原因是现有 PDF.js 阅读器已经覆盖页面渲染、缩放、翻页和错误状态，这些都是当前显示目标仍然需要的能力。相比重新接回旧 iframe 方案，移除批注相关 UI 和事件路径的变更面更小，也保留了以后再次讨论批注时的阅读器基础。

备选方案一是完全回退到原生 iframe 阅读器。它更简单，但会丢失已实现的阅读控制，并把未来任何 PDF.js 方向都推回起点，因此不采用。

备选方案二是保留现有批注代码但隐藏 UI。这样会留下大量不可见但仍然活跃的状态读写和事件逻辑，导致范围不清晰，因此不采用。

### 决策 2：文档内嵌预览保持当前轻量 `embed` 路径

文档中的嵌入 PDF 继续使用当前轻量预览组件，不在本次回退中改造成 PDF.js 组件。当前目标只要求“能显示 PDF”，而不是在文档正文中提供复杂阅读或批注体验；保留现状可以避免编辑器节点、性能和尺寸控制出现额外风险。

### 决策 3：PDF 显示路径必须彻底脱离批注 sidecar 读写

`PdfViewerDialog` 和 `PdfJsViewer` 不再发起 `attachments.pdfState.get/update` 调用，也不再渲染保存批注、冲突重载、批注列表或历史批注提示。显示模式必须只依赖 PDF 文件本身。这样可以确保“只显示 PDF”的产品边界在代码路径上也成立。

备选方案是保留只读批注展示。虽然技术上可行，但仍然会把用户带回“系统支持 PDF 批注”的认知，不符合当前范围，因此不采用。

### 决策 4：`AttachmentPdfState` 作为非当前能力暂时保留，但不再由阅读流程使用

为了降低数据库和后端回退风险，本次设计默认不立即删除 `AttachmentPdfState` 表、模型、路由和历史数据，而是先把它们从当前 PDF 显示功能中完全解耦。这样既能快速收缩用户可见能力，也保留日后重新设计 PDF 批注时可参考的历史实现。

如果实现阶段发现这套代码会持续影响构建、类型或维护成本，可以在同一变更中进一步清理，但前提仍然是“不影响 PDF 显示主路径”。

## 风险 / 权衡

- 保留 PDF.js 而不是完全回退 iframe -> 运行时仍然比最轻量方案更重；缓解：保留懒加载和当前阅读器结构，只删除批注路径。
- 保留 dormant 的 `AttachmentPdfState` 代码和数据 -> 会形成一定技术债；缓解：明确其不属于当前支持能力，并从显示路径彻底断开。
- 批注 UI 移除不彻底 -> 用户可能仍看到“保存批注”“旧批注已隐藏”等残留文案；缓解：对字符串、按钮、API 调用和测试做一次完整审计。
- 回退中误伤 PDF 阅读控制 -> 可能导致缩放、翻页或错误状态回归；缓解：对显示模式的关键行为保留专项测试和构建验证。

## 迁移计划

1. 先从前端阅读界面移除批注侧栏、工具和保存流程。
2. 再从显示路径中移除 `pdfAttachmentState` 相关调用、状态和文案。
3. 保留 PDF.js 的页面渲染、导航和缩放，并验证 worker 和生产构建仍可用。
4. 将 `AttachmentPdfState` 相关服务端能力视为 dormant 状态，不再由当前 UI 调用。
5. 不执行数据库数据迁移；现有记录若存在则继续保留但不参与显示流程。

## 开放问题

- 是否需要在本次回退中顺手删除完全无引用的批注类型和测试辅助代码，还是只要求从主路径脱离即可。
- 如果后续连 PDF.js 也不想保留，是否另起一条更激进的回退变更，把独立阅读界面也退回浏览器原生预览。
