## 为什么

当前 PDF 方案已经扩展到页内批注、侧车状态保存和相关编辑交互，但产品范围现在只需要稳定的 PDF 显示能力。需要把范围收缩到“可靠显示、浏览和打开原文件”，以便先交付明确的阅读体验，并把批注能力留到后续单独讨论。

## 变更内容

- 保留 PDF 的显示能力，包括文档内嵌预览和独立阅读界面中的 PDF 打开与阅读。
- 保留独立阅读界面的基础阅读控制，如页面渲染、滚动、跳页、缩放和打开原文件。
- 移除用户可见的 PDF 批注能力，包括文本高亮、矩形标注、颜色/模式工具、批注列表、删除、保存和冲突提示。
- **BREAKING** `AttachmentPdfState` 及其批注保存流程不再作为当前版本支持的用户可见能力；PDF 显示流程不得再依赖批注状态读写。
- 保持原始 PDF attachment 文件语义不变：打开或下载时仍返回原文件，不生成“批注后的 PDF 文件”。
- 允许临时保留未暴露的内部代码或历史数据，但这些内容不得影响或暴露到 PDF 显示流程。

## 功能 (Capabilities)

### 新增功能

无。

### 修改功能

- `pdfjs-annotation-layer`: 将能力范围从“PDF.js 阅读器 + 页内批注”收缩为“PDF.js 阅读器的显示与浏览”。
- `pdf-attachment-state`: 撤回当前版本中用户可见的 PDF 批注状态读取、展示和保存要求，使 PDF 显示流程独立于侧车批注状态。

## 影响

- 前端：`app/components/PdfViewerDialog.tsx`、`app/components/PdfViewer/*`、`app/utils/pdfAttachmentState.ts` 及相关测试需要收缩到 display-only 路径。
- 共享类型：PDF 批注专用类型和帮助函数需要从当前显示流程中解耦。
- 后端：`AttachmentPdfState` 模型、API 和 schema 需要明确为非当前显示流程依赖，或从活跃功能路径中移除。
- 产品行为：界面文案、按钮和状态提示必须只表达 PDF 阅读，不得暗示存在当前可用的批注保存能力。
