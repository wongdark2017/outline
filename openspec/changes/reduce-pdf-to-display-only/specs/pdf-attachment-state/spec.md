## 新增需求

### 需求:系统必须使 PDF 显示独立于侧车状态
系统在 PDF 显示模式下必须能够在没有 PDF 侧车状态的情况下展示 PDF 内容，并禁止因为批注状态读取、保存或冲突处理影响 PDF 阅读。

#### 场景:无侧车状态时仍可显示 PDF
- **当** 用户打开一个没有任何 PDF 侧车状态的 PDF attachment
- **那么** 系统必须正常显示 PDF 内容

#### 场景:显示模式不触发侧车请求
- **当** PDF 阅读态以显示模式渲染
- **那么** 前端禁止调用 `attachments.pdfState.get` 或 `attachments.pdfState.update`

## 修改需求

### 需求:系统必须明确 PDF 侧车状态可见性
系统必须把任何现存的 PDF 侧车状态视为非当前版本支持的内部数据，并禁止在 PDF 显示流程中向用户暗示存在可用的批注保存能力或原始 PDF 已被修改。

#### 场景:下载原 PDF 不包含侧车状态
- **当** 用户直接打开或下载原始 attachment
- **那么** 系统必须返回原 PDF 文件且不得承诺包含任何批注结果

#### 场景:显示模式不展示批注保存语义
- **当** 前端展示 PDF 阅读界面
- **那么** 系统不得展示“保存批注”“重新加载批注”或等效的批注状态文案

#### 场景:现有侧车状态不影响显示
- **当** 后端已经存在该 PDF 的历史侧车状态记录
- **那么** 当前版本的 PDF 显示流程必须忽略这些记录并继续显示 PDF 内容

## 移除需求

### 需求:系统必须保存 PDF attachment 侧车状态
**Reason**: 当前版本不支持用户可见的 PDF 批注和保存能力。
**Migration**: PDF 阅读流程必须停止依赖或触发侧车状态创建与更新；历史数据可保留但不再作为活跃产品能力。

### 需求:系统必须按文档和附件限定状态作用域
**Reason**: 当前版本不再把 PDF 侧车状态作为支持中的用户能力。
**Migration**: 若相关后端结构暂时保留，它们只能作为 dormant 内部实现存在，不得影响显示流程。

### 需求:系统必须校验 PDF 状态读写权限
**Reason**: 当前版本不再暴露 PDF 批注状态读写流程。
**Migration**: 从当前产品路径中移除对 PDF 状态读写权限的依赖；未来恢复批注能力时必须重新定义并验证权限模型。

### 需求:系统必须防止 PDF 状态并发静默覆盖
**Reason**: 当前版本不再支持用户可见的 PDF 状态更新。
**Migration**: 显示模式不得再呈现 revision 冲突处理；未来恢复写入能力时必须重新定义并验证并发策略。

### 需求:系统必须校验 PDF 状态数据
**Reason**: 当前版本不再把 PDF 状态写入作为支持中的产品行为。
**Migration**: 从当前 UI 和测试中移除对 PDF 批注数据结构的活跃承诺；若后端代码暂时保留，需明确其不属于当前显示功能范围。

### 需求:前端必须以库无关方式展示和保存 PDF 批注状态
**Reason**: 当前版本不再支持批注展示和保存。
**Migration**: 从阅读界面中移除该能力；文档内嵌和独立阅读态都只保留 PDF 显示。

### 需求:前端必须以 PDF.js 页内方式展示和保存 PDF 批注状态
**Reason**: 当前版本不再支持 PDF.js 页内批注展示和保存。
**Migration**: 从 PDF.js 阅读器中移除 `pdfState` 读写、批注渲染和相关 UI；未来如恢复，必须通过新的变更重新定义。
